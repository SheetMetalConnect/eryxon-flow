Killing the whiteboard: a 14-day rollout at Demar Laser.

Demar runs four cells, three shifts, and 1,800 unique parts a year off a TRUMPF laser. Until this spring, the schedule lived on three whiteboards in the planner's office. Here's what changed in two weeks.

LV
Luke van Enkhuizen
Co-founder · Eryxon
· 22 May 2026 · 7 min read rollout sheet metal

Demar's planner kept a whiteboard for each cell. Laser, brake, weld, paint. Each morning she'd walk the floor, rub off completed jobs, write new ones, and call back to her desk with a list of operators to reassign. By Wednesday the whiteboards were a smear of half-erased part numbers. By Friday she was redoing Monday's planning from scratch.

This is the most common MES failure mode we see at small fabricators. Not "operators won't use the tablet" — that one's solved. The real failure is that the planner ends up doing double work: once on the whiteboard, once in whatever system management bought. So they stop using the system. The whiteboard stays.

We approached the Demar rollout with that failure mode in mind. Three rules:

  1. The planner's screen is the source of truth. No parallel whiteboard, no backup spreadsheet.
  2. The operators see the same state, live. If the planner reassigns a job at 09:14, the tablet at the brake reflects it by 09:14:01.
  3. Day one is one cell. Not "the floor". One cell, one shift.

Day one: the laser cell only

We picked the laser cell first because it's where every job starts — if the laser sees the new system, the whole shop eventually does. Demar had been pushing jobs in from Oseon already, so the data was clean. We imported one week of WIP and one week of upcoming work via the REST API.

The laser operator — Jonas, 14 years with Demar, sceptical of every digital tool brought in before — used the tablet for the morning. By lunchtime he'd asked us to remove two filters from the queue toolbar because they were "stuff the planner cares about". We did.

i
Lesson: cut the operator UI ruthlessly. The planner and the operator do not want the same screen. Don't try to give them one. The operator wants three buttons and a status stripe; the planner wants a table.

The API push from Oseon

Demar's existing Oseon installation already had work orders, parts, and operations. We didn't want them re-keyed. The bridge we set up was small — eight endpoints on our side, one cron on theirs. Here's the core of it:

// /functions/oseon-sync/handle-job-push.ts
export async function POST(req: Request) {
  const auth = await verifySignature(req);
  if (!auth.ok) return new Response("bad signature", { status: 401 });

  const jobs = await req.json();
  const normalised = jobs.map(toEryxonJob);

  return upsertJobs(normalised, { tenant: auth.tenant });
}

Two things to note. First, signatures: every push from Oseon is HMAC-signed, so we can verify it came from Demar's installation without a long-lived API key on their side. Second, idempotency: upsertJobs uses Oseon's job_number as the unique key, so re-pushing yesterday's batch doesn't create duplicates. The cron runs every 90 seconds.

"We tried this with our last MES. They told us to 'expose the data' and gave us a CSV importer. Eryxon showed up with a working bridge to Oseon on day four."

— Luc Vermeer, operations manager, Demar Laser

Week two: the brake, the weld, the rest

Once the laser was running clean — about three days in — we added the brake. Then welding the day after that. Each cell took less time than the last because the operators in the new cell could see the old cells already living on Eryxon and asked the natural questions ("how do I clock in?", "what's the orange tag mean?") to the operators next door instead of to us.

By day twelve the four cells were on. The planner started skipping the morning whiteboard walk. By day fourteen the whiteboards came down.

✓
Rollout outcome — measured at day 30. 38 % fewer "what's next" calls between planner and floor. Planner's daily replanning time down from ~75 min to ~25 min. No measurable change in operator throughput (yet) — operators tell us things feel calmer.

What we'd do differently

Two things, both small. One: we should have set up the issue queue on day one, not day eight. Demar's QA found three bend-tolerance issues in the first week that lived on a paper form and never made it to the system. That's data we'd want.

And two: we under-sold the customisation. Every shop has a "this is how we do it" detail — Demar wanted operator initials on every card, the next shop will want the cell name in the URL, the one after that will want a sound on overdue jobs. Build for these and stop treating them as edge cases.

If you're rolling out an MES this quarter, start with one cell. Then call us.
