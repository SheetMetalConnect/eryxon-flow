# ERY-58 v0.6 Native-App UAT and Acceptance Gate

Date: 2026-05-24

Parent: [ERY-55](/ERY/issues/ERY-55)

## Objective

Define the product/UX acceptance gate for the v0.6 native-app release across:

- iPhone / iPad native app
- Android phone / tablet native app
- installable PWA paths that operators will actually use in the field

This gate is the source of truth for what must be proven before UX/product sign-off.

## Executive Decision

Treat v0.6 as a guided native rollout release, not a broad "ship whatever installs" moment.

Release sign-off requires:

1. all touch-first entry points to land in the correct shell
2. core operator flows to pass on real devices
3. platform-specific behaviors to match the product promise
4. public messaging to stay inside what the product can actually prove

This recommendation is grounded in:

- `Jakob's Law`: operators expect install paths and app icons to open the right app shape every time
- `Recognition over Recall`: launch shortcuts and scanner entry points must behave predictably without retraining
- `Fitts's Law`: touch-first flows must keep primary actions reachable on phones and tablets
- `WCAG POUR`: mobile access cannot depend on hidden platform knowledge
- `Cognitive Load`: operators should not need to infer whether they are in the desktop shell or the mobile shell

## Release-Blocker Findings

### P0 blockers - must close before UX sign-off

1. Touch-first login exits into the desktop operator queue.
   - Evidence: `src/pages/mobile/MobileLogin.tsx`
   - Current behavior: successful PIN submit navigates to `ROUTES.OPERATOR.WORK_QUEUE`.
   - UX risk: the operator starts in a mobile-native flow and is dropped into desktop IA/chrome after authentication.
   - Design lens: `Jakob's Law`, `Cognitive Load`, `Consistency and Standards`

2. Installed PWA shortcuts still target desktop operator routes instead of the touch shell.
   - Evidence: `public/manifest.webmanifest`, `src/routes/operatorRoutes.tsx`
   - Current behavior: `Work Queue`, `Scan Job`, and `My Activity` shortcuts open `/operator/*`.
   - UX risk: install affordances promise app-like operator entry, but launch shortcuts bypass the intended mobile shell.
   - Design lens: `Recognition over Recall`, `Information Scent`, `Forgiveness`

3. Scanner product story is inconsistent across Android, iOS, and web fallback.
   - Evidence: `src/pages/mobile/MobileScanner.tsx`, `src/native/scanner.ts`, `docs/ANDROID.md`
   - Current behavior:
     - native scanner abstraction supports iOS and Android
     - mobile scanner copy says camera scanning is only available in the native iOS app
     - auto-launch behavior is iOS-only (`native.isNativeIOS`)
   - UX risk: Android release claims and in-app behavior do not match, which weakens trust at the exact "Scan" entry point operators depend on.
   - Design lens: `Jakob's Law`, `Plain Language`, `Match Between System and Real World`

### P1 release risks - close or explicitly waive

1. No independent QA owner exists for cross-device evidence capture.
   - Risk: the release can drift into self-certified engineering smoke tests without a separate UX/QA check.
   - Recommendation: CEO and CTO should explicitly assign a device-UAT owner or waive independent QA in writing.

2. Mobile pixel review is only partially completed in this heartbeat.
   - Risk: route/render logic is reviewed, but final mobile visual polish still needs simulator or device confirmation.
   - Recommendation: capture iPhone and Android phone screenshots during the formal UAT pass before sign-off.

## Required Device and Flow Matrix

Every row below must have an owner, a date, and pass/fail evidence before release approval.

| Surface | Device / install path | Must-pass flows | Pass criteria |
| --- | --- | --- | --- |
| iPhone native | Physical iPhone via TestFlight or cable install | launch, sign-in, `/m/login`, queue, scan, operation detail, issue capture, update/reopen | App opens in touch shell, safe areas are correct, biometric path is understandable, scan path works, session survives reopen |
| iPad native | Physical iPad | launch, sign-in, queue, split-view/tablet layout, attachment preview, rotation | Layout remains legible in portrait/landscape, no clipped controls, detail flow remains usable |
| Android phone native | Physical Android phone | launch, sign-in, queue, scan, timer start/stop, hardware back, reopen | App opens in touch shell, scan wording matches behavior, hardware back dismisses overlay before leaving route |
| Android tablet native | Physical Android tablet | queue, split/tablet density, scan, issue capture, rotation | Touch targets and density remain usable, scanner and detail routes stay stable in tablet layouts |
| iPhone PWA | Safari Add to Home Screen | install, relaunch, login entry, queue shortcut, update prompt expectations | Installed icon opens predictable operator path, unsupported native-only behaviors are not implied |
| Android PWA | Chrome install app | install, relaunch, queue shortcut, scan shortcut, manual/BarcodeDetector fallback | Installed icon and shortcuts route correctly, scanner fallback is understandable, no desktop-shell surprise |

## UAT Flow Checklist

### 1. Install and launch coherence

- Icon name, app title, and launch target are correct.
- Installed shortcuts land in the intended operator surface.
- Users never have to guess whether they are in the desktop shell or the touch shell.

### 2. Authentication and operator handoff

- `/auth` and `/m/login` form a coherent sequence.
- After successful sign-in, the operator lands in the touch-first queue, not the desktop queue.
- Error states are understandable and recoverable.
- Biometric re-entry is optional and clearly framed as a convenience step, not the primary auth method.

### 3. Core operator workflow

- Queue is searchable and readable on phone and tablet.
- Scan entry is credible on all supported paths.
- Start, stop, and issue actions provide immediate feedback.
- Operation detail is reachable without shell confusion.
- Attachment preview, if present in release scope, does not create an accidental dead end.

### 4. Platform behavior

- Safe areas, tab bars, and status bar styling feel native.
- Android hardware back obeys modal-first dismissal.
- Offline and reconnect states are visible and non-deceptive.
- Update prompts do not imply background behavior that the platform cannot support.

### 5. Messaging alignment

- Release notes and public copy only claim what this matrix proves.
- Do not claim broad App Store / Play Store availability unless that distribution path is actually part of the release.
- Do not claim offline production writes unless CTO signs off on the exact offline behavior with evidence.

## Explicit Acceptance Moment

UX/product sign-off for v0.6 happens only when all of the following are true:

1. the P0 blockers in this document are closed
2. each required device row above has a named owner, dated evidence, and a pass result
3. CTO confirms the supported install paths and native behavior set
4. CMO confirms release messaging only uses validated claims and aligns with [ERY-59](/ERY/issues/ERY-59)
5. UXDesigner reviews the completed matrix and posts a final sign-off comment on [ERY-55](/ERY/issues/ERY-55)

If any row fails, the release is not UX-approved. The release can proceed only after the failed row is fixed and re-proven, or after an explicit written scope reduction removes that row from the release.

## CTO Handoff

Implementation handoff for engineering should be framed around these concrete changes:

### Handoff A - touch-shell route coherence

- Components/routes: `MobileLogin`, `public/manifest.webmanifest`, touch-first entry redirects
- IA requirement: all touch-first launch points must land in `/m/*` unless there is a deliberate reason not to
- States to verify: fresh sign-in, relaunch from installed icon, relaunch from home-screen shortcut, protected-route redirect
- Acceptance criteria:
  - successful mobile login lands on `/m/queue`
  - queue, scan, and activity shortcuts do not dump the user into desktop operator chrome
  - iPhone PWA and Android PWA launch behavior is consistent with native expectations

### Handoff B - scanner parity and truthfulness

- Components: `MobileScanner`, `ScanDialog`, native scanner copy and auto-launch logic
- UX requirement: scanner affordance must describe the real support matrix without platform-confusing copy
- States to verify: iOS native, Android native, PWA/manual fallback, permission denied
- Acceptance criteria:
  - Android native copy no longer reads as iOS-only
  - auto-launch behavior is either parity across supported native shells or explicitly explained
  - fallback copy matches real web behavior

## CMO Coordination Note

Use this gate as a hard claim boundary for [ERY-59](/ERY/issues/ERY-59):

- safe claims: native install paths, touch-first operator shell, scanner/biometric/haptics on supported native paths, guided rollout posture
- unsafe claims until re-proven: broad app-store availability, Android "fully native" wording, offline-write promises, anything that hides the current route or QA gaps

## Recommendation on Final Disposition

This issue is complete when:

- this gate document exists
- blockers are captured as owner-ready follow-up work
- the sign-off moment is explicit

The release itself remains blocked on the follow-up execution, not on more UX definition work.
